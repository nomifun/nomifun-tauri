# 新建知识库桌面端滚动修复设计

## 背景与根因

桌面端的新建知识库弹窗仅通过 `max-height` 限制外层高度，没有建立确定的可用高度。弹窗主体在桌面布局下又切换为 CSS Grid；其隐式行默认按内容高度扩张，导致右侧配置区计算出的滚动范围大于屏幕上真正可见的区域。窗口缩小时，外层的 `overflow-hidden` 会裁切主体，表现为滚动条已经到底但底部内容仍不可见；窗口尺寸变化后，滚动条还可能因重新布局而消失。

Web 端窗口通常较高，因此不容易触发该布局边界，但两端实际共用同一套弹窗代码。

## 方案取舍

1. 让整个弹窗（包括头部和底部操作区）一起滚动：实现简单，但创建按钮会离开视区，降低小窗口下的可达性。
2. 给弹窗建立确定的动态视口高度，并让中间 Grid 行可收缩：推荐。头部和底部操作区保持固定，仅中间配置区滚动，符合现有交互。
3. 使用 JavaScript 监听窗口尺寸并手工计算高度：可以解决问题，但会引入额外状态、监听器和同步时序，不适合此处纯布局问题。

## 设计

- 桌面端弹窗内容高度使用 `min(760px, calc(100dvh - 80px))`：保持现有最大视觉高度，同时在窗口缩小时立即收缩。
- 移动端使用 `calc(100dvh - 48px)`，为弹窗内容区的上下内边距预留空间，并使用动态视口单位适配窗口尺寸变化。
- 桌面端主体 Grid 显式设置 `grid-template-rows: minmax(0, 1fr)`，右列改为 `minmax(0, 1fr)`，允许主体和配置列在受限空间内收缩。
- 右侧配置面板补充 `min-h-0`，使其成为唯一的纵向滚动容器。
- 不修改表单状态、提交逻辑、文案、数据接口或来源类型组件。

## 验收与验证

1. 结构回归测试断言弹窗具有确定的动态视口高度，Grid 行可收缩，配置面板拥有正确的滚动约束。
2. 先确认测试在现状下失败，再实施最小布局修改并确认测试通过。
3. 运行知识库页面相关测试、UI 类型检查和生产构建。
4. 桌面端缩小窗口后打开新建知识库，滚动到底时底部教学内容完整可见，固定底部操作区不遮挡内容；再次改变窗口大小后滚动条仍按内容正确出现。
