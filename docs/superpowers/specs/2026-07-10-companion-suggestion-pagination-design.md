# 桌面伙伴建议列表分页设计

## 目标

为“桌面伙伴 / 共享 / 建议”提供与记忆列表一致的服务端分页。建议数量增长后，用户仍能清楚查看各状态下的建议，并在采纳或忽略时保留稳定、可预测的列表位置。

## 范围

- 将 `GET /api/companion/suggestions` 由数组返回改为包含当前页与筛选总数的页面对象。
- 为建议列表增加默认 10 条、可切换 20 或 50 条的分页控件。
- 保持现有建议卡片、状态筛选与“采纳 / 忽略”业务语义不变。

不增加搜索、排序、批量决策或新的建议字段。

## 接口与数据流

接口成功载荷为：

```ts
interface ICompanionSuggestionPage {
  items: ICompanionSuggestion[];
  total: number;
}
```

请求继续使用 `status`，新增 `offset`，并以 `limit` 指定页大小。服务端对页面查询和 `COUNT(*)` 使用同一 `status` 条件，排序固定为 `created_at DESC`。`limit` 仍限制在 1–500，负数 `offset` 按 0 处理。

前端维护 1 基 `page`、`pageSize` 与 `total`。默认 `pageSize` 为 10，允许 10 / 20 / 50；请求参数为 `limit: pageSize` 与 `offset: (page - 1) * pageSize`。切换任意状态筛选或每页数量都回到第 1 页。

采纳、忽略以及实时创建/决策事件刷新当前查询。若决策移除了当前页最后一张建议卡，使当前页超出新的最大页码，页面自动切换至最后一个有效页再请求。

## 列表与反馈

保留现有卡片结构：建议类别图标、标题、说明、创建时间及仅“待处理”状态可用的采纳/忽略操作。卡片区域在后续页加载时保留上一页内容并降低透明度；首次加载显示现有加载态。无结果时保留现有空状态。

当筛选结果大于零时，页脚显示 Arco Pagination 的总数、页码、上一页/下一页、按需跳页与每页数量选择。只有一页时仍显示页大小选择，方便用户主动增加或减少阅读密度。

## 验收与测试

- 存储层测试验证指定状态下的总数、页大小和偏移量一致，且其他状态不计入总数。
- 桥接测试验证页面对象与 `offset` 参数；建议页测试验证默认 10 条、10/20/50 选择和状态重置分页。
- `cargo test -p nomifun-companion --lib`、相关 Bun 测试、`bun run typecheck` 与 `git diff --check` 均通过。

## 设计自检

范围仅覆盖建议的读取分页与页脚交互；没有改变建议生成、决策副作用或状态定义。页面查询、总数、前端状态和回退逻辑均使用相同的状态筛选条件，避免跨状态计数或空页。
