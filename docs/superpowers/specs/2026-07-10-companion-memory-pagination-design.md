# 桌面伙伴记忆列表分页与信息层级设计

## 目标

让“桌面伙伴 / 记忆”在记忆数量增长后仍保持可快速扫描、可预测操作和稳定加载：默认每页 10 条，允许用户切换至 20 或 50 条，并以服务端分页避免全量加载。

## 范围

- 为 `GET /api/companion/memories` 增加筛选后的总数，并保留现有 `limit`、`offset` 的分页语义。
- 前端根据页码、页大小和现有筛选条件请求单页数据。
- 重整记忆行的视觉层级与操作密度；新增、编辑、置顶、归档和删除的业务语义不变。

不引入批量选择、排序切换或新的记忆字段。

## 接口与数据流

`GET /api/companion/memories` 的成功载荷由记忆数组改为：

```ts
interface ICompanionMemoryPage {
  items: ICompanionMemory[];
  total: number;
}
```

服务端以同一组 `kind`、`q`、`status`、`scope_companion_id` 条件分别查询当前页和总数，排序固定为置顶优先、强度降序、最近更新降序。`limit` 限制仍为 1–500，`offset` 小于 0 时按 0 处理。

前端维持 `page`（从 1 开始）和 `pageSize`（默认 10）。请求参数为 `limit: pageSize` 与 `offset: (page - 1) * pageSize`。类别、状态、可见范围和关键词的每一次变更都会重置到第 1 页。页面大小变更同样回到第 1 页。新增、编辑、置顶、归档、删除及实时事件均刷新当前结果；若删除导致当前页超出最后一页，则重新请求最后一个有效页。

## 列表与分页交互

筛选栏保持类别、状态、范围、搜索和“添加记忆”的位置与能力。列表上方显示筛选后的总条数。

每条记忆使用两层信息：

1. 首行显示类别标签、置顶标识及最多两行的记忆正文。
2. 次行显示可见范围、强度、更新时间和非学习来源。

置顶保持为可发现的图标按钮；编辑、归档/恢复和删除收拢进“更多”菜单，删除项为危险样式并保留二次确认。这样保留高频控制，同时避免每行重复的文字按钮淹没正文。

页脚使用现有 Arco `Pagination`：显示总数、页码、上一页/下一页、按需跳页和 10/20/50 页大小选择。翻页时保留当前行并降低其透明度，同时显示轻量加载状态，避免列表区域闪为空白；首次加载使用与当前密度相近的骨架行。

## 错误与空状态

请求失败显示可读错误，并保留上一次成功页以供继续浏览。筛选无结果显示空状态和“添加记忆”入口。筛选结果总数为零时不显示无效分页控件。

## 验收与测试

- 后端测试验证：筛选后的 `items` 和 `total` 使用同一条件；`limit/offset` 正确返回页面；排序不变。
- 前端测试验证：首屏为第 1 页 10 条；切换页码/页大小使用正确请求参数；修改筛选重置第 1 页；超页删除回退；分页控件接收总数和可选页大小。
- `ui` 类型检查与相关测试均通过。

## 设计自检

本设计没有未决项、占位实现或额外功能。接口、数据流和 UI 均围绕同一套筛选条件；总数来自服务端，避免因前端截断或数据增长产生错误页数。范围限于记忆列表，不改变记忆写入或权限模型。
